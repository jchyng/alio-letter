# 구체적 구현 계획

## 기술 스택

| 구분        | 선택                                     | 이유                                                    |
| ----------- | ---------------------------------------- | ------------------------------------------------------- |
| 자동화 엔진 | n8n (라즈베리파이)                       | 워크플로우 기반, 노코드/로우코드로 전체 파이프라인 구성 |
| DB          | SQLite 3                                 | 서버리스, 소규모 데이터에 적합, n8n에서 직접 접근       |
| **PDF 파싱**| **Gemini 1.5 Flash API (Direct)**        | **n8n HTTP Request 노드로 직접 호출, 가장 깔끔한 연동** |
| 스크래핑    | n8n HTTP Request + Code 노드             | 잡알리오 공고 목록/상세 크롤링                          |
| 사용자 관리 | Google Form + Google Sheets              | 가입/스펙입력/수정을 Google 생태계로 처리               |
| 알림 발송   | 이메일 (Gmail 노드) → 추후 카카오 알림톡 | 우선 이메일, 사용자 증가 시 알림톡 추가                 |
| 결제        | 계좌이체 (수동 확인)                     | 초기 단계에서 간단하게 운영                             |

> **💡 파싱 전략: API 기반 구조화 추출**
> n8n 내부에서 별도의 CLI 설치 없이 **Google AI API 전용 엔드포인트**를 호출하여 PDF를 직접 전송. (1.5 Flash 모델 사용으로 비용과 속도 최적화)

## Gemini API 비용 분석 및 최적화

### 예상 비용 (Gemini 1.5 Flash 기준)
- **계산 방식**: PDF 1페이지당 258토큰으로 계산 (Gemini의 멀티모달 처리 방식)
- **40페이지 기준**: 40 × 258 = 10,320 tokens + 프롬프트 텍스트
- **단가**: $0.075 / 1,000,000 tokens (입력 토큰 기준)
- **결과**: 파일 1개당 약 **$0.0009 (한화 약 1.3원)**
- **월 예상 운영비**: 하루 20개 공고 수집 시, 월 약 800원 내외

### 모델 선택 전략
1. **Gemini 1.5 Flash (확정)**: 
   - 높은 처리 속도와 파격적인 저비용
   - 잡알리오 공고의 표/구조 데이터 추출에 가장 적합
2. **Gemini 1.5 Pro (백업)**:
   - 복합적인 자격 요건 분석이 매우 복잡하여 Flash가 실패할 때만 사용 (비용 약 15배 이상)

## 시스템 아키텍처

```
[사용자]
  ├── Google Form 작성 (가입 + 스펙 + 희망조건)
  ├── Google Form 수정 (수정 링크로 재접근)
  └── 이메일로 분석 결과 수신

[Google Cloud / API]
  ├── Google Sheets (데이터 저장소)
  ├── Gmail (알림 발송)
  └── Gemini 1.5 Flash API (PDF 구조화 분석 및 요약)

[라즈베리파이 - n8n]
  ├── 워크플로우 1: 공고 수집 (Schedule → 스크래핑 → PDF 다운로드 → Gemini API → SQLite)
  ├── 워크플로우 2: 사용자 동기화 (Sheets Trigger → SQLite 동기화)
  ├── 워크플로우 3: 매칭 + 알림 (신규공고 감지 → 매칭 → 이메일 발송)
  └── SQLite DB 파일 (alio.db)
```

## 워크플로우 상세 설계

### 워크플로우 1: 공고 수집 파이프라인

**트리거:** Schedule (매일 오전 9시, 오후 6시)

```
[Schedule Trigger]
    ↓
[HTTP Request] 잡알리오 전체 공고 목록 페이지 조회
    ↓
[Code 노드] HTML 파싱 → 공고 링크 추출
    ↓
[Code 노드] DB에서 마지막 수집 공고 조회 → 신규 공고만 필터링 (Stack 생성)
    ↓
[Loop Over Items] Stack 순서대로 반복
    ↓
  ├── [HTTP Request] 상세 페이지 접근 → 메타데이터 추출
  ├── [HTTP Request] PDF 다운로드 (바이너리 데이터 확보)
  ├── [Read Binary File] 프롬프트 파일(`prompt.md`) 로컬에서 읽기
  ├── [HTTP Request] Gemini API 호출 (PDF + 프롬프트 → JSON 추출)
  ├── [Code 노드] JSON 검증 (인원 합계 체크, 필수 필드)
  ├── [IF] 검증 통과?
  │     ├── YES → [SQLite] 구조화 데이터 분산 저장 + status='active'
  │     └── NO  → [SQLite] status='review_needed' 표시
  ├── [HTTP Request] Gemini API 호출 (PDF 요약 생성)
  └── [SQLite] 요약 저장
    ↓
[워크플로우 3 트리거] 신규 active 공고에 대해 매칭 실행
```

## 배포 환경

| 배포 환경 | 내용                                    |
| -------- | --------------------------------------- |
| 하드웨어 | 라즈베리파이 (n8n 상시 가동)            |
| n8n      | Docker 컨테이너 (Node.js 기반)          |
| **데이터 유지** | **호스트-컨테이너 볼륨 마운트 (`-v` 옵션) 사용** |
| SQLite   | 호스트의 `~/n8n-data/alio.db` (컨테이너 내 `/data/alio.db`로 마운트) |
| PDF/프롬프트| 호스트의 `~/n8n-data/assets/` (컨테이너 내 `/data/assets/`로 마운트) |
| 외부 API | Gemini API (API Key 인증)               |
| 모니터링 | n8n 실행 이력 대시보드 (내장)           |

> **Note: 데이터 지속성(Persistence) 확보**
> 모든 중요 파일(DB, PDF, 프롬프트)은 라즈베리파이 호스트 디렉토리에 저장하고 Docker 볼륨 마운트를 통해 n8n 컨테이너와 공유함. 이를 통해 컨테이너 재시작/업데이트 시에도 데이터 손실 방지 및 호스트에서의 손쉬운 백업 가능.
