# 핵심 로직 플로우

## 한 줄 요약

잡알리오 채용공고 파일을 AI가 읽어 핵심 정보를 구조화하고 요약 → 사용자 스펙과 자동 매칭 → 지원 가능 여부와 가점 체크리스트까지 제공

<br>

## 핵심 데이터

### 1. 사용자 관심 조건 (필터)

> 사용자가 등록 시 입력하는 관심 조건.
> **잡알리오 검색 필터와 동일한 분류 체계**를 사용하여, 공고 메타데이터와 직접 매칭한다.
> 빈 값(NULL/빈 배열)은 "전체"로 간주하여 해당 항목 필터링을 건너뛴다.

#### `users` 테이블

| 컬럼명                  | 타입                          | 필수 | 설명                                     |
| ----------------------- | ----------------------------- | ---- | ---------------------------------------- |
| user_id                 | INT                           | PK   | AUTO_INCREMENT                           |
| name                    | VARCHAR(50)                   | O    | 이름                                     |
| email                   | VARCHAR(100)                  | O    | 이메일 (알림 발송 주소)                  |
| is_active               | TINYINT(1)                    | O    | 활성 상태 (1: 활성, 0: 비활성), 기본값 1 |
| subscription_status     | ENUM('free','paid','expired') | O    | 구독 상태, 기본값 'free'                 |
| subscription_expires_at | DATE                          | -    | 구독 만료일 (paid일 때만)                |
| created_at              | DATETIME                      | O    | 가입일, 기본값 NOW()                     |
| updated_at              | DATETIME                      | O    | 수정일, 기본값 NOW() ON UPDATE           |

#### `user_preferences` 테이블

| 컬럼명                | 타입 | 필수  | 설명                             |
| --------------------- | ---- | ----- | -------------------------------- |
| user_id               | INT  | PK/FK | users.user_id 참조               |
| pref_regions          | JSON | -     | 희망 근무지역 (복수선택)         |
| pref_job_fields       | JSON | -     | 희망 근무분야 (복수선택)         |
| pref_employment_types | JSON | -     | 희망 고용형태 (복수선택)         |
| pref_recruit_types    | JSON | -     | 희망 채용구분 (복수선택)         |
| pref_education_levels | JSON | -     | 학력정보 (복수선택)              |
| pref_org_types        | JSON | -     | 관심 기관유형 (복수선택)         |
| pref_org_names        | JSON | -     | 관심 기관명 (복수선택, 선택사항) |

#### 필터 항목별 가능한 값 (잡알리오 기준)

| 필터 항목             | 가능한 값                                                                                                                                                                                            |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| pref_regions          | 해외, 서울특별시, 인천광역시, 대전광역시, 대구광역시, 부산광역시, 광주광역시, 울산광역시, 경기도, 강원도, 충청남도, 충청북도, 경상북도, 경상남도, 전라남도, 전라북도, 제주특별자치도, 세종특별자치시 |
| pref_job_fields       | 일반직, 행정직, 경영직, 사무직, 관리직, 연구직, 기술직, 시설직, 전산직, 교수직, 의사직, 간호직, 약무직, 보건직, 위생직, 기능직, 운영직, 공무직, 안전직, 교원직, 별정직, 계약직, 전문직, 상담직, 기타 |
| pref_employment_types | 정규직, 무기계약직, 비정규직, 청년인턴(체험형), 청년인턴(채용형)                                                                                                                                     |
| pref_recruit_types    | 신입, 경력, 신입+경력, 외국인전형                                                                                                                                                                    |
| pref_education_levels | 학력무관, 중졸이하, 고졸, 대졸(2~3년), 대졸(4년), 석사, 박사                                                                                                                                         |
| pref_org_types        | 공기업, 준정부기관, 기타공공기관, 지방공기업, 지방출자출연기관                                                                                                                                       |
| pref_org_names        | 자유 입력 (예: 한국남부발전, 한국전력공사, 한국수자원공사 등) => 잡알리오에서 가져와서 똑같이 select + 검색 가능하게 하면 됨                                                                         |

#### 매칭 로직

```
공고 메타데이터의 각 항목이 사용자 필터 JSON 배열에 포함되면 매칭.
필터가 비어있으면(NULL 또는 빈 배열) → 해당 항목은 "전체"로 간주 → 무조건 매칭.

예시: 한국남부발전 공고 (부산광역시, 기술직, 정규직, 신입, 대졸(4년), 공기업)
→ 사용자A: pref_regions=["부산광역시","서울특별시"], pref_job_fields=["기술직"] → 매칭 O
→ 사용자B: pref_regions=["서울특별시"], pref_job_fields=["전산직"] → 매칭 X (지역, 분야 모두 불일치)
→ 사용자C: pref_regions=[], pref_job_fields=["기술직"] → 매칭 O (지역 전체 + 분야 일치)
```

### 2. 공고 데이터 (메타데이터 + Gemini 파싱)

> 데이터가 **2개 레이어**로 나뉜다.
> - **`postings`**: 잡알리오 목록 페이지에서 수집하는 메타데이터 → 사용자 필터 매칭에 사용
> - **`posting_tracks`**: 공고문 PDF를 Gemini로 파싱한 상세 데이터 → 자격 체크 + 가점 계산에 사용
>
> **1개 공고 = N개 채용 트랙** 관계. (예: 한국남부발전 1개 공고 안에 대졸일반, 대졸장애, 대졸보훈, 고졸, 별정직 = 5개 트랙)

#### `postings` 테이블 — 잡알리오 메타데이터

> 잡알리오 목록/상세페이지에서 수집. user_preferences와 동일한 분류 체계로 저장하여 매칭 쿼리를 단순화한다.

| 컬럼명 | 타입 | 필수 | 설명 |
|---|---|---|---|
| posting_id | INT | PK | AUTO_INCREMENT |
| alio_id | VARCHAR(20) | UQ | 잡알리오 공고 고유번호 |
| title | VARCHAR(200) | O | 공고 제목 |
| org_name | VARCHAR(100) | O | 기관명 |
| org_type | VARCHAR(50) | O | 기관유형 |
| regions | JSON | O | 근무지역 (잡알리오 분류) |
| job_fields | JSON | O | 근무분야 (잡알리오 분류) |
| employment_types | JSON | O | 고용형태 |
| recruit_types | JSON | O | 채용구분 |
| education_levels | JSON | O | 학력정보 |
| positions_summary | JSON | - | 모집분야별 전체 인원 요약 (Gemini 파싱) |
| total_positions | INT | - | 총 채용인원 (Gemini 파싱) |
| posting_url | VARCHAR(500) | O | 잡알리오 상세페이지 URL |
| file_urls | JSON | - | 공고문 파일 URL 목록 |
| posted_date | DATE | O | 공고 등록일 |
| deadline | DATETIME | O | 접수 마감일시 |
| schedule | JSON | - | 채용일정 (Gemini 파싱) |
| status | ENUM('진행중','마감') | O | 공고 상태, 기본값 '진행중' |
| created_at | DATETIME | O | DB 등록일시, 기본값 NOW() |

#### `posting_tracks` 테이블 — Gemini 파싱 결과

> 공고문 PDF를 Gemini API로 파싱하여 **채용 트랙별**로 저장.
> 자격 조건 체크와 가점 계산의 입력 데이터가 된다.

| 컬럼명 | 타입 | 필수 | 설명 |
|---|---|---|---|
| track_id | INT | PK | AUTO_INCREMENT |
| posting_id | INT | FK | postings.posting_id 참조 |
| track_name | VARCHAR(50) | O | 채용유형명 |
| grade | VARCHAR(30) | - | 채용직급 |
| positions | JSON | O | 모집분야별 인원 |
| total_positions | INT | O | 해당 트랙 총 채용인원 |
| work_locations | JSON | - | 근무지역 상세 |
| work_type | VARCHAR(50) | - | 근무형태 |
| eligibility | JSON | O | 지원자격 조건 |
| selection_process | JSON | O | 전형절차 및 배점 |
| scoring_criteria | JSON | - | 서류심사 점수 산출 기준 |
| bonus_points | JSON | - | 가점 항목 및 전형별 배점률 |
| certificate_bonus_table | JSON | - | 자격증 가점 세부 기준 |
| language_conversion_table | JSON | - | 어학성적 환산 기준 |
| quota_policies | JSON | - | 채용목표제 (이전지역인재, 양성평등 등) |

#### `posting_tracks` JSON 필드 상세 구조

각 JSON 필드의 구조를 한국남부발전 **"대졸수준-일반"** 트랙 기준으로 예시한다.

**① positions — 모집분야별 인원**

```json
[
  {"field": "사무", "count": 8},
  {"field": "ICT", "count": 7},
  {"field": "발전기계", "count": 25},
  {"field": "발전전기", "count": 21},
  {"field": "화학", "count": 7},
  {"field": "토목", "count": 4},
  {"field": "건축", "count": 7}
]
```

**② eligibility — 지원자격**

```json
{
  "age": "제한 없음 (만 60세 미만)",
  "education": "제한 없음",
  "language": "제한 없음",
  "certificate": "제한 없음",
  "military": "병역의무 불이행자에 해당하지 않는 자. 현역의 경우 최종합격자 발표일 이전에 전역 가능한 자",
  "special": null,
  "required_conditions": [],
  "preferred_conditions": []
}
```

> 트랙별 차이 예시:
> - 대졸-장애: `"special": "장애인고용촉진 및 직업재활법에 의한 장애인 등록자"`
> - 대졸-보훈: `"special": "국가유공자 등 예우 및 지원에 관한 법률에 의한 취업지원대상자"`
> - 고졸: `"education": "최종학력이 고등학교 졸업 또는 졸업예정인 자 (전문대·대학 졸업자 지원 불가)"`
> - 별정직: `"required_conditions": ["지원 모집단위 근무지역(하동) 출퇴근 가능자"]`, `"preferred_conditions": ["폐기물·환경·가스 관련 자격증 보유자 (폐기물처리기술사, 수질환경기사 등)"]`

### 3. 사용자 스펙 정보 (자격 체크 + 가점 계산 입력)

> 사용자가 등록 시 입력하는 본인 스펙 정보.
> `posting_tracks`의 자격 조건·가점·자격증 가점·어학 환산·채용목표제를 체크하기 위한 **입력 데이터**다.
> 자격증만 1:N 관계이므로 별도 테이블(`user_certificates`)로 분리한다.

#### `user_specs` 테이블

| 컬럼명 | 타입 | 필수 | 설명 |
|---|---|---|---|
| user_id | INT | PK/FK | users.user_id 참조 |
| birth_date | DATE | O | 생년월일 (나이 제한 체크용) |

> **어학성적 유효기간 처리**: 공고 마감일 기준으로 만료 여부를 자동 체크한다.
> 성적이 여러 시험에 걸쳐 있으면 `language_conversion_table`로 TOEIC 환산 후 **최고점**을 적용한다.

> **가점 boolean 필드 설계 이유**: `bonus_points`의 가점 항목들이 단순 적부 판정(해당/비해당)이므로 boolean 플래그로 충분하다. 각 공고마다 가점 항목이 다를 수 있지만, **사용자 스펙은 고정값**이고 매칭 시점에 해당 공고의 `bonus_points` 목록과 대조하면 된다.

#### `master_certificates` 테이블 — 자격증 마스터 데이터

> 사용자가 자격증을 **검색·선택**하는 기준 테이블.
> "정처기"처럼 줄임말을 입력하는 문제를 원천 차단하고, 분야·등급 매핑을 자동화한다.
> 데이터 소스: 한국산업인력공단 국가자격 목록 + 공고별 `certificate_bonus_table`에서 누적 수집.

| 컬럼명 | 타입 | 필수 | 설명 |
|---|---|---|---|
| master_cert_id | INT | PK | AUTO_INCREMENT |
| cert_name | VARCHAR(100) | UQ | 공식 자격증명 (예: 정보처리기사) |
| cert_grade | VARCHAR(30) | O | 등급 (기술사, 기사, 산업기사, 기능사, 1급, 2급, 3급 등) |
| cert_field | JSON | O | 해당 분야 목록 (예: ["ICT"] 또는 ["전기","기계"]) |
| cert_type | VARCHAR(20) | O | 자격 유형 (국가기술, 국가전문, 민간, 어학, 한국사 등) |

> **cert_field를 JSON 배열로 하는 이유**: 일부 자격증(예: 산업안전기사)은 기계·전기·화학 등 여러 분야에서 가점이 인정된다.
> `certificate_bonus_table.by_field`의 키와 직접 매핑되므로, 가점 계산 시 분야 교차 검증이 가능하다.

#### `user_certificates` 테이블

| 컬럼명 | 타입 | 필수 | 설명 |
|---|---|---|---|
| cert_id | INT | PK | AUTO_INCREMENT |
| user_id | INT | FK | users.user_id 참조 |
| master_cert_id | INT | FK | master_certificates.master_cert_id 참조 |
| issue_date | DATE | O | 취득일 |

> 사용자는 자격증명을 타이핑하지 않는다. **검색 → 선택** 방식으로 `master_certificates`에서 고른다.
> `cert_name`, `cert_grade`, `cert_field`는 모두 `master_cert_id` JOIN으로 가져온다.
>
> **가점 계산 흐름**:
> 1. `user_certificates.master_cert_id` → `master_certificates`에서 `cert_name`, `cert_grade`, `cert_field` 조회
> 2. 해당 공고의 `certificate_bonus_table.by_field[cert_field]`에서 `cert_grade` 매칭 → 점수 산출
> 3. `certificate_bonus_table.common`에서 공통 자격증 매칭 → 점수 산출
> 4. 동일종류 상위등급 1개만 인정, 최대 3개까지 합산 (규칙은 가점 계산 로직에서 처리)

#### 가점 계산 매핑 — `user_specs` ↔ `posting_tracks.bonus_points`

```
posting_tracks.bonus_points.type    →  user_specs 컬럼
─────────────────────────────────────────────────────
등록장애인                          →  is_disabled = 1
취업지원대상자                      →  is_veteran = 1
고급자격증                          →  user_certificates에서 기사 이상 보유 여부
발전소주변지역주민                  →  residence_detail (발전소 소재지 매칭)
저소득층                            →  is_low_income = 1
북한이탈주민                        →  is_north_korean_defector = 1
다문화가족                          →  is_multicultural_family = 1
자립준비청년                        →  is_independent_youth = 1
체험형인턴수료자                    →  kospo_intern_status = '수료'
체험형인턴우수수료자                →  kospo_intern_status = '우수수료'
시간선택제근로자                    →  is_part_time_worker = 1
경력단절여성                        →  is_career_break_woman = 1
```

#### 자격 체크 매핑 — `user_specs` ↔ `posting_tracks.eligibility`

```
posting_tracks.eligibility 항목    →  user_specs 컬럼
─────────────────────────────────────────────────────
age (나이 제한)                    →  birth_date에서 공고 마감일 기준 만 나이 계산
education (학력 요건)              →  education_level + education_status
military (병역 요건)               →  military_status + military_discharge_date
special (장애/보훈 요건)           →  is_disabled / is_veteran + 각 type 필드
required_conditions (필수 조건)    →  케이스별 판단 (거주지, 학력제한 등)
preferred_conditions (우대 조건)   →  user_certificates 등에서 해당 여부 체크
```

#### 채용목표제 매핑 — `user_specs` ↔ `posting_tracks.quota_policies`

```
quota_policies 항목                →  user_specs 컬럼
─────────────────────────────────────────────────────
이전지역인재                       →  school_region (학교 소재지가 본사 소재지와 일치)
이공계 양성평등                    →  gender + major_category (이공계 여부)
```

## 핵심 기능

### 공고 데이터 수집 플로우

1. DB에서 가장 마지막으로 등록된 공고 번호(last_no) 조회 → `postings.alio_id` MAX 값
2. 잡알리오 사이트(https://job.alio.go.kr/recruit.do?ing=2)에서 가장 최신 공고부터 last_no까지 각 공고의 메타데이터와 상세페이지 링크 정보 Stack에 저장
3. Stack에서 하나씩 뽑으면서 해당 공고 상세페이지로 이동
4. 상세페이지에서 아래 정보를 수집하여 `postings` 테이블에 저장:
   - `alio_id`, `title`, `org_name`, `org_type`
   - `regions`, `job_fields`, `employment_types`, `recruit_types`, `education_levels` (잡알리오 분류 메타데이터)
   - `posting_url`, `file_urls` (공고문 파일 URL 목록)
   - `posted_date`, `deadline`
   - `status` = '진행중'
5. 수집한 공고문 파일(이미지, hwp, hwpx, pdf)을 Gemini API로 분석하여 **트랙별** 아래 정보를 추출 → `posting_tracks` 테이블에 저장:
   - `track_name`, `grade`, `positions`, `total_positions`, `work_locations`, `work_type`
   - `eligibility` (나이, 학력, 어학, 자격증, 병역, 특수조건, 필수조건, 우대조건)
   - `selection_process` (전형절차 및 배점)
   - `scoring_criteria` (서류심사 점수 산출 기준 — 어학 배점, 자격증 배점)
   - `bonus_points` (가점 항목 목록 및 전형별 배점률)
   - `certificate_bonus_table` (분야별·등급별 자격증 가점 세부 기준)
   - `language_conversion_table` (TOEIC Speaking·OPIc → TOEIC 환산표)
   - `quota_policies` (채용목표제 — 이전지역인재, 양성평등 등)
6. Gemini 파싱 결과에서 공고 전체 요약 정보도 `postings`에 업데이트:
   - `positions_summary` (모집분야별 인원 요약)
   - `total_positions` (총 채용인원)
   - `schedule` (채용일정)

### 공고 자동 분석 및 결과 알림

1. 새로 수집한 공고의 메타데이터(`postings`)로, `user_preferences`를 대조하여 매칭되는 사용자 조회:
   - `postings.regions` ∩ `user_preferences.pref_regions`
   - `postings.job_fields` ∩ `user_preferences.pref_job_fields`
   - `postings.employment_types` ∩ `user_preferences.pref_employment_types`
   - `postings.recruit_types` ∩ `user_preferences.pref_recruit_types`
   - `postings.education_levels` ∩ `user_preferences.pref_education_levels`
   - `postings.org_type` ∈ `user_preferences.pref_org_types`
   - `postings.org_name` ∈ `user_preferences.pref_org_names`
   - (빈 값은 "전체"로 간주하여 무조건 매칭)
2. 매칭된 사용자가 1명이라도 있으면, 각 사용자별로 Gemini API에 아래 데이터를 전달하여 **자격 체크 + 가점 계산** 수행:
   - **공고 측**: 해당 공고의 `posting_tracks` 전체 (eligibility, bonus_points, certificate_bonus_table, language_conversion_table, scoring_criteria, quota_policies)
   - **사용자 측**: `user_specs` (생년월일, 학력, 병역, 어학성적, 가점 플래그 전체) + `user_certificates` (보유 자격증 목록 → `master_certificates` JOIN)
   - **산출 결과**: 트랙별 지원 가능 여부, 자격 미충족 항목, 서류 예상 점수(어학+자격증), 적용 가능 가점 목록, 채용목표제 해당 여부
3. 수행된 결과를 DB에 사용자별·공고별 분석 결과 테이블에 저장
4. 1~3을 새로운 공고가 없을 때까지 반복
5. 사용자의 분석 결과 중, 발송된 적 없는 내용을 사용자 이메일(`users.email`)로 발송
6. 해당 결과의 발송 상태를 업데이트
