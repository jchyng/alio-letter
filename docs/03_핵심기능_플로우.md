# 핵심 기능 플로우

## 전체 흐름 개요

```
[스케줄러] 매일 오전 7시
    │
    ▼
[FLOW 1] 공고 수집
잡알리오 크롤링 → 공고 저장 → Gemini로 핵심 정보 추출 → DB 저장
    │
    ▼
[FLOW 2] 사용자 매칭
새 공고 × 모든 유저의 관심 조건과 검사 → 관심 있는 사용자 목록 추출
    │
    ▼
[FLOW 3] 자격 체크 & 가점 계산
사용자가 지원할 수 있는 채용 구분 추출 → 각 채용 조건 중 → user_posting_analyses 저장
    │
    ▼
[FLOW 4] 이메일 발송
분석 결과 → 이메일 구성 → 발송 → 발송 시각 기록
```

<br>

## FLOW 1 — 공고 수집

### 1-1. 신규 공고 목록 수집

1. DB에서 `postings.alio_id` 최댓값(last_no) 조회
2. 잡알리오 목록 페이지(`/recruit.do?ing=2`)를 최신순으로 순회하며 `alio_id > last_no`인 공고 수집
3. 각 공고의 메타데이터를 스택에 적재:
   - `alio_id`, `title`, `org_name`, `org_type`
   - `regions`, `job_fields`, `employment_types`, `recruit_types`, `education_levels`
   - `posting_url`, `file_urls`, `posted_date`, `deadline`
4. 공고 상세 페이지의 `연봉보러가기` 버튼을 스크래핑하여 DB(`salary_info`)에 해당 기관 연봉 정보가 없을 경우 실시간으로 수집 및 저장.
5. `postings` 테이블에 저장, `status = '진행중'`

### 1-2. 공고문 파싱 (Gemini)

> **역할 분리**
>
> - **스크래퍼 담당**: `posting_url`, `file_urls`, `alio_id`, `title`, `org_name` 등 목록/상세 페이지에서 수집 가능한 메타데이터 → 1-1에서 이미 저장 완료
> - **Gemini 담당**: PDF/HWP 비구조화 문서 → 구조화 JSON 변환만. `posting_url`, `file_urls`는 Gemini 출력에 포함하지 않음
> - **자격 체크·점수 계산**: FLOW 3에서 코드로 처리 (Gemini 미사용)

5. 수집된 공고의 `file_urls`에서 파일 다운로드 (PDF, HWP, HWPX, 이미지)
6. Gemini API로 파싱하여 트랙별 구조화 데이터 추출 → `posting_tracks`에 저장:
   - `track_name`, `grade`, `positions`, `total_positions`, `work_locations`, `work_type`
   - `eligibility` (나이, 학력, 어학, 자격증, 병역, 특수조건, 필수/우대조건)
   - `selection_process` (전형절차 및 배점, 동점자 처리 기준 포함)
   - `scoring_criteria` (서류 어학 배점 공식, 유효기간 조건, 자격증 배점 한도)
   - `bonus_points` (가점 항목·자격 조건 원문·전형별 배점률)
   - `bonus_points_rule` (가점 중복 한도 등 운영 규칙)
   - `certificate_bonus_table` (분야별·등급별 자격증 가점 세부 기준, 교차 가점 제한 규칙 포함)
   - `language_conversion_table` (TOEIC Speaking·OPIc → TOEIC 환산표)
   - `quota_policies` (채용목표제 — 이전지역인재, 양성평등 등)
     > **주의**: 트랙 간 내용이 같더라도 Gemini는 반드시 각 트랙의 데이터를 완전히 독립 출력.
     > "동일" 참조 문자열이 감지되면 서버측 검증에서 `parse_status = 'failed'`로 처리.
7. `postings` 테이블에 요약 정보 업데이트:
   - `positions_summary` (모집분야별 인원 요약)
   - `total_positions` (총 채용인원)
   - `schedule` (채용일정)

### 1-3. 파싱 실패 처리

| 상황                              | 처리                                                                                   |
| --------------------------------- | -------------------------------------------------------------------------------------- |
| 파싱 성공                         | `posting_tracks` 정상 저장, `parse_status = 'done'`                                    |
| 파싱 결과 불완전 (필수 필드 누락) | `parse_status = 'partial'`, 가능한 데이터만 저장, 알림 발송 시 "일부 정보 미확인" 표시 |
| 파싱 실패 (파일 오류, API 에러)   | `parse_status = 'failed'`, FLOW 3 건너뜀, 공고 원문 링크만 이메일에 포함               |
| 파일 없음 (file_urls = [])        | `parse_status = 'no_file'`, 동일하게 원문 링크만 발송                                  |

> `parse_status`는 `postings` 테이블에 관리.

## FLOW 2 — 사용자 매칭

새로 수집된 공고 1건에 대해 매칭되는 사용자를 조회한다.

```
조건: 각 항목에서 교집합이 존재하면 매칭. 사용자 필터가 빈 값(NULL/빈 배열)이면 해당 항목은 "전체"로 간주.

postings.location        ∩  users.filter_prefs.locations
postings.work_field      ∩  users.filter_prefs.work_fields
postings.employment_type ∩  users.filter_prefs.employment_types
postings.recruit_type    ∩  users.filter_prefs.recruit_types
postings.education       ∩  users.filter_prefs.educations
postings.org_type        ∈  users.filter_prefs.org_types
postings.org             ∈  users.filter_prefs.org_names
```

> `users.filter_prefs`는 JSON 컬럼. 빈 값(null 또는 빈 배열)이면 해당 항목 "전체"로 간주.
> 매칭된 사용자 중 활성 상태인 사용자만 대상으로 FLOW 3 진입.

<br>

## FLOW 3 — 자격 체크 + 가점 판정

> Gemini가 공고 원문(`posting_tracks.eligibility`, `bonus_points`)과 사용자 스펙(`users.parsed_spec`)을 함께 보고 판정.
> 코드 기반 재현 가능성이 필요한 항목은 Python으로 직접 처리.

매칭된 사용자 × 공고의 트랙 조합마다 아래를 수행.

### 3-1. 자격 체크

`posting_tracks.eligibility` 항목 × `users.parsed_spec` (UserProfile) 대조:

| eligibility 항목 | 체크 방법 |
| ---------------- | --------- |
| `education`      | `parsed_spec.education` 비교 |
| `career`         | `parsed_spec.career_years` + `career_fields` 비교 |
| `age`            | `parsed_spec.birth_year` 기준 공고 `deadline` 시점 만 나이 계산 |
| `language`       | `parsed_spec.languages` 비교 |
| `certificate`    | `parsed_spec.certificates` 비교 |
| `etc`            | 병역, 기타 조건 항목별 판단 |

결과: **지원 가능 여부(`eligible`)** + **미충족 항목 목록(`unmet`)**

### 3-2. 가점 판정

`postings.bonus_points` 원문 × `users.parsed_spec` 대조:

```
판정 항목            →  parsed_spec 필드
────────────────────────────────────────────────
취업지원대상자       →  veteran_type (≠ "해당없음")
등록장애인           →  disability_grade (≠ "해당없음")
저소득층             →  is_low_income = true
북한이탈주민         →  is_north_korean_defector = true
다문화가족자녀       →  is_multicultural_child = true
자립준비청년         →  is_independent_youth = true
```

결과: **가산점 합계 요약(`bonus_summary`)** + **항목별 근거(`bonus_reasons`)**

### 3-3. 결과 저장

위 3-1 ~ 3-2 결과를 `user_judgments`에 저장:

- 트랙별 지원 가능 여부(`eligible`), 미충족 항목(`unmet`)
- 가산점 요약(`bonus_summary`), 항목별 근거(`bonus_reasons`)

> **향후 확장**: 서류 점수 예상(어학·자격증 환산), 채용목표제 체크는 미구현. `posting_tracks.notes`에 Gemini가 기록한 중요 정보 활용 예정.

<br>

## FLOW 4 — 이메일 발송

1. `user_judgments`에서 미발송 결과 조회
2. 사용자별로 분석 결과를 묶어 이메일 1통 구성:
   - 공고 제목, 기관명, 마감일, 초임 연봉, 필기시험 일정
   - 트랙별 지원 가능 여부 (적부 체크 시각화)
   - 서류 예상 점수 (어학 + 자격증, 계산 근거 및 선발 배수 포함)
   - 적용 가능 가점 목록 (점수 합산과 분리하여 표기)
   - 채용목표제 해당 여부 (할당제임을 명시)
   - 직접 확인 필요 항목 (기업별 고유 가점, parse_status = partial/failed인 경우)
   - 잡알리오 상세페이지 링크 + 지원 바로가기
3. 발송 완료 후 발송 이력 기록

<br>

## 실행 스케줄

| 작업                    | 주기             | 비고                                   |
| ----------------------- | ---------------- | -------------------------------------- |
| FLOW 1 공고 수집        | 매일 오전 6시    | 잡알리오 신규 공고 크롤링              |
| FLOW 2~3 매칭·분석      | FLOW 1 완료 직후 | 신규 공고에 대해서만 실행              |
| FLOW 4 이메일 발송      | 매일 오전 7시    | 분석 완료된 결과 일괄 발송             |
| 마감 공고 상태 업데이트 | 매일 오전 0시    | `deadline < NOW()` → `status = '마감'` |
