# 핵심 기능 플로우

## 한 줄 요약

잡알리오 채용공고를 자동 수집·파싱 → 사용자 관심 조건으로 필터링 → 코드로 자격 체크 + 가점 계산 → 이메일 발송

> 데이터 스키마 상세: [04_데이터_스키마.md](./04_데이터_스키마.md)

<br>

## 전체 흐름 개요

```
[스케줄러] 매일 오전 7시
    │
    ▼
[FLOW 1] 공고 수집
잡알리오 크롤링 → postings 저장 → Gemini PDF 파싱 → posting_tracks 저장
    │
    ▼
[FLOW 2] 사용자 매칭
새 공고 × user_preferences → 관심 있는 사용자 목록 추출
    │
    ▼
[FLOW 3] 자격 체크 + 가점 계산
사용자 스펙 × posting_tracks → 트랙별 분석 결과 → user_posting_analyses 저장
    │
    ▼
[FLOW 4] 이메일 발송
미발송 분석 결과 → 이메일 구성 → 발송 → 발송 시각 기록
```

<br>

## FLOW 1 — 공고 수집

### 1-1. 신규 공고 목록 수집

1. DB에서 `postings.alio_id` 최댓값(last_no) 조회
2. 잡알리오 목록 페이지(`/recruit.do?ing=2`)를 최신순으로 순회하며 `alio_id > last_no`인 공고 수집
3. 각 공고의 메타데이터를 스택에 적재:
   - `alio_id`, `title`, `org_name`, `org_type`
   - `regions`, `job_fields`, `employment_types`, `recruit_types`, `education_levels`
   - `posting_url`, `file_urls`, `posted_date`, `deadline`
4. `postings` 테이블에 저장, `status = '진행중'`

### 1-2. 공고문 파싱 (Gemini)

> **역할 분리**: Gemini는 비구조화 문서(PDF/HWP) → 구조화 JSON 변환만 담당.
> 자격 체크·점수 계산은 FLOW 3에서 코드로 처리한다.

5. 수집된 공고의 `file_urls`에서 파일 다운로드 (PDF, HWP, HWPX, 이미지)
6. Gemini API로 파싱하여 트랙별 구조화 데이터 추출 → `posting_tracks`에 저장:
   - `track_name`, `grade`, `positions`, `total_positions`, `work_locations`, `work_type`
   - `eligibility` (나이, 학력, 어학, 자격증, 병역, 특수조건, 필수/우대조건)
   - `selection_process` (전형절차 및 배점)
   - `scoring_criteria` (서류 어학 배점 공식, 자격증 배점 한도)
   - `bonus_points` (가점 항목 및 전형별 배점률)
   - `certificate_bonus_table` (분야별·등급별 자격증 가점 세부 기준)
   - `language_conversion_table` (TOEIC Speaking·OPIc → TOEIC 환산표)
   - `quota_policies` (채용목표제 — 이전지역인재, 양성평등 등)
7. `postings` 테이블에 요약 정보 업데이트:
   - `positions_summary` (모집분야별 인원 요약)
   - `total_positions` (총 채용인원)
   - `schedule` (채용일정)

### 1-3. 파싱 실패 처리

| 상황 | 처리 |
|---|---|
| 파싱 성공 | `posting_tracks` 정상 저장, `parse_status = 'done'` |
| 파싱 결과 불완전 (필수 필드 누락) | `parse_status = 'partial'`, 가능한 데이터만 저장, 알림 발송 시 "일부 정보 미확인" 표시 |
| 파싱 실패 (파일 오류, API 에러) | `parse_status = 'failed'`, FLOW 3 건너뜀, 공고 원문 링크만 이메일에 포함 |
| 파일 없음 (file_urls = []) | `parse_status = 'no_file'`, 동일하게 원문 링크만 발송 |

> `parse_status`는 `postings` 테이블에 관리.

<br>

## FLOW 2 — 사용자 매칭

새로 수집된 공고 1건에 대해 매칭되는 사용자를 조회한다.

```
조건: 각 항목에서 교집합이 존재하면 매칭. 사용자 필터가 빈 값(NULL/빈 배열)이면 해당 항목은 "전체"로 간주.

postings.regions              ∩  user_preferences.pref_regions
postings.job_fields           ∩  user_preferences.pref_job_fields
postings.employment_types     ∩  user_preferences.pref_employment_types
postings.recruit_types        ∩  user_preferences.pref_recruit_types
postings.education_levels     ∩  user_preferences.pref_education_levels
postings.org_type             ∈  user_preferences.pref_org_types
postings.org_name             ∈  user_preferences.pref_org_names
```

> 매칭된 사용자 중 `users.is_active = 1`인 사용자만 대상으로 FLOW 3 진입.

<br>

## FLOW 3 — 자격 체크 + 가점 계산

> 코드로 직접 계산하여 결과의 정확성과 재현 가능성을 보장한다.

매칭된 사용자 × 공고의 트랙 조합마다 아래를 수행.

### 3-1. 자격 체크

`posting_tracks.eligibility` 항목 × `user_specs` 대조:

| eligibility 항목 | 체크 방법 |
|---|---|
| `age` | `birth_date` 기준 공고 `deadline` 시점 만 나이 계산 |
| `education` | `education_level` + `education_status` 비교 |
| `military` | `military_status` + `military_discharge_date` (현역 시 전역 가능 여부) |
| `special` | `is_disabled` / `is_veteran` + 각 type 필드 |
| `required_conditions` | 항목별 판단 (거주지, 학력 제한 등) |
| `preferred_conditions` | `user_certificates` 등에서 해당 여부 체크 |

결과: **지원 가능 여부** + **미충족 항목 목록**

### 3-2. 서류 점수 계산

`posting_tracks.scoring_criteria` + `posting_tracks.certificate_bonus_table` × `user_specs` / `user_certificates`:

**어학 점수 계산**
1. `user_specs`의 TOEIC, TOEIC Speaking, OPIc 성적 중 공고 마감일 기준 유효한 것만 사용
2. TOEIC Speaking·OPIc는 `language_conversion_table`로 TOEIC 점수로 환산
3. 환산된 최고점으로 `scoring_criteria.language_score.formula` 적용 → 어학 점수 산출
4. 어학 성적이 없으면 0점

**자격증 점수 계산**
1. `user_certificates` 목록 → `master_certificates` JOIN으로 `cert_name`, `cert_grade`, `cert_field` 조회
2. `certificate_bonus_table.common`에서 공통 자격증 점수 매칭
3. `certificate_bonus_table.by_field`에서 지원 분야별 자격증 점수 매칭
4. 동일종류 상위등급 1개만 인정, 최대 3개 합산 (규칙은 `certificate_bonus_table.rules` 준수)

### 3-3. 가점 계산

`posting_tracks.bonus_points` × `user_specs` 대조:

```
bonus_points.type          →  user_specs 컬럼
────────────────────────────────────────────────
등록장애인               →  is_disabled = 1
취업지원대상자           →  is_veteran = 1
고급자격증               →  user_certificates에서 기사 이상 보유 여부
발전소주변지역주민       →  residence_detail (발전소 소재지 매칭)
저소득층                 →  is_low_income = 1
북한이탈주민             →  is_north_korean_defector = 1
다문화가족               →  is_multicultural_family = 1
자립준비청년             →  is_independent_youth = 1
경력단절여성             →  is_career_break_woman = 1
시간선택제근로자         →  is_part_time_worker = 1
```

> 공고에 위 목록 외의 기업별 고유 가점 항목이 있으면, 해당 항목은 이메일에 "직접 확인 필요"로 표시.

### 3-4. 채용목표제 체크

```
quota_policies 항목     →  user_specs 컬럼
────────────────────────────────────────────────
이전지역인재            →  school_region이 공고 기관 본사 소재지와 일치
이공계 양성평등         →  gender + major_category (이공계 여부)
```

### 3-5. 결과 저장

위 3-1 ~ 3-4 결과를 `user_posting_analyses`에 저장:
- 트랙별 지원 가능 여부, 미충족 항목
- 서류 예상 점수 (어학 + 자격증)
- 적용 가능 가점 목록 및 전형별 배점률
- 채용목표제 해당 여부
- 이메일 발송 상태 (`email_sent_at = NULL`)

<br>

## FLOW 4 — 이메일 발송

1. `user_posting_analyses`에서 `email_sent_at IS NULL`인 결과 조회
2. 사용자별로 분석 결과를 묶어 이메일 1통 구성:
   - 공고 제목, 기관명, 마감일
   - 트랙별 지원 가능 여부 (적부 체크 시각화)
   - 서류 예상 점수 (어학 + 자격증, 계산 근거 포함)
   - 적용 가능 가점 목록
   - 채용목표제 해당 여부
   - 직접 확인 필요 항목 (기업별 고유 가점, parse_status = partial/failed인 경우)
   - 잡알리오 상세페이지 링크 + 지원 바로가기
3. 발송 완료 후 `user_posting_analyses.email_sent_at` 업데이트

<br>

## 실행 스케줄

| 작업 | 주기 | 비고 |
|---|---|---|
| FLOW 1 공고 수집 | 매일 오전 6시 | 잡알리오 신규 공고 크롤링 |
| FLOW 2~3 매칭·분석 | FLOW 1 완료 직후 | 신규 공고에 대해서만 실행 |
| FLOW 4 이메일 발송 | 매일 오전 7시 | 분석 완료된 결과 일괄 발송 |
| 마감 공고 상태 업데이트 | 매일 오전 0시 | `deadline < NOW()` → `status = '마감'` |
