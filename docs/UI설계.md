# UI 설계 문서 — Alio-Letter 웹 페이지

> Cloudflare Pages 기반 정적 사이트 (Phase 1: 사용자 등록 시스템)
> 총 5개 페이지: 랜딩 / 회원등록 / 프로필 수정 / 결제 / 결제 완료

---

## 페이지 구조 전체 맵

```
/                  → 랜딩 페이지 (서비스 소개)
/login             → 내 정보 관리 (간편 로그인 링크 요청)
/register          → 회원등록 폼
/profile/:token    → 등록정보 및 구독 관리 (마이페이지)
/subscribe/:token  → 구독 결제 페이지 (토스페이먼츠)
/subscribe/done    → 결제 완료/실패 페이지
```

### 공통 사항

| 항목 | 내용 |
|------|------|
| 호스팅 | Cloudflare Pages (정적 HTML) |
| 백엔드 | Cloudflare Functions (Gemini API + MySQL 직접 연결) |
| 반응형 | 모바일 우선 (취준생은 모바일 이용률 높음) |
| 디자인 톤 | 깔끔하고 신뢰감 있는 톤. 공공기관 취업 도우미 느낌 |
| 폰트 | Pretendard (또는 시스템 폰트 fallback) |
| 주 색상 | 파란 계열 (#2563EB) — 신뢰/안정 |
| 보조 색상 | 초록 (#16A34A) — 성공/완료, 빨강 (#DC2626) — 오류/경고 |

### 공통 레이아웃

```
┌─────────────────────────────────────┐
│  [로고] Alio-Letter      [등록하기] │  ← 상단 네비게이션
├─────────────────────────────────────┤
│                                     │
│           페이지 콘텐츠             │  ← 각 페이지별 내용
│                                     │
├─────────────────────────────────────┤
│  © 2026 Alio-Letter                 │  ← 하단 푸터
│  출처: 잡알리오(job.alio.go.kr)     │
│  문의: 이메일주소                   │
└─────────────────────────────────────┘
```

## 1. 랜딩 페이지 (`/`)

### 목적
서비스가 무엇인지 빠르게 이해시키고, 회원등록으로 자연스럽게 유도한다.

### 섹션 구성

#### 1-1. 히어로 섹션
- **헤드라인**: "공기업 채용공고, 더 이상 직접 찾지 마세요"
- **서브 카피**: "잡알리오 채용공고를 AI가 분석해서, 내 스펙에 맞는 공고만 이메일로 받아보세요"
- **상단 네비게이션 추가**: `[내 정보 관리]` 버튼 (클릭 시 이메일 입력 팝업 노출 -> 가입한 이메일로 관리 링크 재발송 기능)
- **CTA 버튼**: `[무료로 시작하기]` → `/register`로 이동
- **비주얼**: 이메일 리포트 미리보기 이미지 (실제 이메일 템플릿 스크린샷)

#### 1-2. 핵심 가치 제안 (3가지)
| 아이콘 | 제목 | 설명 |
|--------|------|------|
| 🔍 | 자동 공고 탐색 | 잡알리오 신규 채용공고를 매일 자동으로 수집합니다 |
| 🤖 | AI 맞춤 분석 | 내 스펙 기반 지원자격 검토 + 가점 분석까지 한눈에 |
| 📩 | 이메일 리포트 | 분석 결과를 깔끔한 리포트로 이메일에 전달합니다 |

#### 1-3. 서비스 작동 흐름 (How it works)
```
Step 1: 내 스펙과 희망조건을 등록
      ↓
Step 2: AI가 매일 잡알리오 공고를 분석
      ↓
Step 3: 내 조건에 맞는 공고만 이메일로 수신
```
각 스텝을 시각적 다이어그램 또는 번호 카드 형태로 표현

#### 1-4. 이메일 리포트 미리보기
실제 이메일 템플릿의 축소 스크린샷을 보여주며, 어떤 정보가 포함되는지 설명:
- AI 요약
- 매칭된 포지션 정보
- 자격 요건 체크 결과 (✅/❌)
- 서류 점수 예상
- 가점 항목

#### 1-5. 베타 테스트 안내 (무료)
현재 베타 테스트 기간으로 모든 기능을 **무료**로 제공하고 있습니다.
- **제공 기능**: AI 맞춤 공고 분석, 서류 점수 예측, 가점 분석 등 전체 기능
- **안내 문구**: "베타 기간 동안 모든 기능이 무료입니다. 지금 등록하고 AI 맞춤 공고 분석을 경험해보세요!"

#### 1-6. FAQ (자주 묻는 질문)
| 질문 | 답변 |
|------|------|
| 정말 무료인가요? | 네, 현재 베타 테스트 기간으로 모든 서비스를 무료로 이용하실 수 있습니다. |
| 어떤 공고를 분석하나요? | 잡알리오(job.alio.go.kr)에 등록되는 모든 공공기관 채용공고를 대상으로 합니다 |
| 알림은 얼마나 자주 오나요? | 매일 2회(오전/오후) 신규 공고를 탐색하며, 매칭되는 공고가 있을 때만 이메일을 보냅니다 |
| 내 정보는 안전한가요? | 입력하신 스펙 정보는 공고 매칭 목적으로만 사용되며 제3자에게 공유되지 않습니다 |
| 등록 후 정보 수정은 어떻게 하나요? | 등록 확인 이메일에 포함된 수정 링크로 언제든 변경할 수 있습니다 |

#### 1-7. 하단 CTA
- **문구**: "공부에만 집중하세요. 공고 분석은 저희가 할게요."
- **버튼**: `[지금 등록하기]` → `/register`

## 2. 내 정보 관리 페이지 (`/login`)

### 목적
이미 등록한 사용자가 자신의 정보를 수정하거나 구독을 관리하기 위해 '관리용 매직 링크'를 요청하는 전용 페이지이다.

### 구성 요소
- **이메일 입력창**: 가입 시 사용한 이메일 주소 입력.
- **링크 발송 버튼**: 클릭 시 `/api/send-magic-link` 호출.
- **성공 메시지**: "입력하신 이메일로 링크가 발송되었습니다. 이메일함을 확인해주세요."

## 3. 회원등록 페이지 (`/register`)

### 목적
사용자의 기본정보, 스펙, 희망조건을 수집한다.
자유서술 형태로 입력받아 Gemini가 구조화 파싱한다.

### 폼 필드 상세

#### 2-1. 기본정보 영역

| 필드명 | 입력 타입 | 필수 | 유효성 검증 | 설명 |
|--------|-----------|------|-------------|------|
| 이름 | `<input type="text">` | O | 2~20자 한글/영문 | |
| 이메일 | `<input type="email">` | O | 이메일 형식, 중복 체크 | 알림 수신 + 계정 식별 키 |

#### 2-2. 나의 스펙 영역

| 필드명 | 입력 타입 | 필수 | 설명 |
|--------|-----------|------|------|
| 나의 스펙 | `<textarea>` | O | 자유서술 형태 |

**입력 안내 문구 (placeholder 또는 도움말 박스):**
```
예시) 4년제 전기공학 졸업, 전기기사/소방설비기사 보유,
토익 820점, 토익스피킹 Lv7, 군필, 등록장애인 해당

💡 아래 내용을 자유롭게 적어주세요:
• 최종 학력 + 전공 (예: 4년제 전기공학과 졸업)
• 생년월일 또는 나이 (선택. 연령 제한 체크에 사용)
• 보유 자격증
• 어학 성적 (토익, 토익스피킹, OPIc 등)
• 병역 상태 (군필/미필/면제/해당없음)
• 해당되는 항목: 등록장애인, 취업지원대상자(보훈),
  저소득층, 북한이탈주민, 다문화가족, 자립준비청년
```

**Gemini가 파싱할 구조화 필드:**
- `education_level`: 최종학력 (고졸/전문대졸/대졸/석사이상)
- `education_major`: 전공/학과명 (예: "전기공학", "경영학", "기계공학" 등. 언급 없으면 null)
- `birth_year`: 출생연도 (4자리 정수, 예: 1998. 언급 없으면 null. 나이로 말한 경우 역산)
- `military_status`: 병역 (미필/군필/면제/해당없음)
- `toeic_score`: 토익 점수
- `toeic_speaking_level`: 토익스피킹 레벨
- `opic_level`: OPIc 레벨
- `certificates`: 보유 자격증 배열
- `is_disabled`: 등록장애인 여부
- `is_veteran_family`: 취업지원대상자 여부
- `is_low_income`: 저소득층 여부
- `is_north_defector`: 북한이탈주민 여부
- `is_multicultural`: 다문화가족 여부
- `is_self_reliance`: 자립준비청년 여부
- `residence_region`: 거주 지역

#### 2-3. 희망 조건 영역

| 필드명 | 입력 타입 | 필수 | 설명 |
|--------|-----------|------|------|
| 희망 조건 | `<textarea>` | O | 자유서술 형태 |

**입력 안내 문구 (placeholder 또는 도움말 박스):**
```
예시) 전기 또는 기계 직무, 서울/경기/부산 근무 희망,
한전/한수원/가스공사 같은 에너지 공기업 관심,
일반채용 또는 장애인채용, 연봉 5천만원 이상

💡 아래 내용을 자유롭게 적어주세요:
• 희망 직무 분야 (사무, ICT, 기계, 전기, 화학, 토목, 건축 등)
• 희망 근무 지역 (시/도 단위)
• 관심 기관 이름
• 희망 채용구분 (일반, 장애, 보훈, 고졸 등)
• 희망 연봉 수준 (선택. 예: 5천만원 이상)
```

**Gemini가 파싱할 구조화 필드:**
- `preferred_fields`: 희망 직무 배열
- `preferred_regions`: 희망 지역 배열
- `preferred_types`: 희망 채용구분 배열
- `preferred_institutions`: 관심 기관 배열
- `preferred_min_salary`: 희망 최소 연봉 (만원 단위 정수, 예: 5000. 언급 없으면 null)

#### 2-4. 폼 하단

- **제출 버튼**: `[등록하기]`
- **개인정보 동의 체크박스**: "입력하신 정보는 공고 매칭 목적으로만 사용됩니다" (필수 체크)
- **안내**: "등록 완료 후 확인 이메일이 발송됩니다"

### UX 동작 흐름

```
[사용자 폼 작성]
    ↓ 제출 버튼 클릭
[클라이언트 기본 검증]
    - 이름: 빈 값 체크
    - 이메일: 형식 검증
    - 스펙/희망조건: 빈 값 체크
    - 동의 체크박스 확인
    ↓ 통과
[로딩 상태 표시] "등록 처리 중..." (스피너 + 버튼 비활성화)
    ↓ fetch('/api/register', POST)
[Cloudflare Function] → Gemini 파싱 → MySQL 저장 (via Tunnel) → 이메일 발송
    ↓ 응답 수신
[성공 시]
    - 성공 메시지 표시: "등록이 완료되었습니다!"
    - 안내: "입력하신 이메일로 확인 메일이 발송되었습니다.
             이메일에 포함된 링크로 언제든 정보를 수정할 수 있습니다."
    - 폼 초기화

[실패 시]
    - 에러 메시지 표시 (인라인)
    - "이미 등록된 이메일입니다" / "서버 오류가 발생했습니다" 등
    - 폼 입력값 유지 (재입력 방지)
```

### 유의사항
- 제출 후 Gemini 파싱까지 시간이 걸릴 수 있으므로 (5~10초), 로딩 인디케이터를 반드시 표시
- 이메일 중복 시: 기존 데이터를 업데이트 (UPSERT) 하되, "이미 등록된 이메일입니다. 정보가 업데이트됩니다." 메시지 표시
- textarea 최소 높이를 충분히 잡아서, 자유서술 입력이 편하도록 함 (최소 120px)

## 3. 프로필 수정 페이지 (마이페이지) (`/profile/:token`)

### 목적
등록 확인 이메일의 수정 링크 또는 **랜딩 페이지의 [내 정보 관리]를 통한 간편 로그인 링크**를 통해 접근한다.
기존 스펙/희망조건을 수정하는 기능뿐만 아니라, 구독 상태 확인 및 결제 관리를 수행하는 **종합 대시보드(마이페이지)** 역할을 한다.

### 진입 경로
- 등록 확인 이메일 본문의 **[내 정보 수정하기]** 링크 클릭
- 랜딩 페이지 **[내 정보 관리]**에서 이메일 입력 후 발송된 '간편 로그인 링크' 클릭
- URL 형식: `https://alio-letter.pages.dev/profile/a1b2c3d4-e5f6-...`
- token은 사용자별 고유 UUID (edit_token)

### 페이지 동작 흐름

```
[페이지 로드]
    ↓ URL에서 token 추출
[데이터 요청] fetch(`/api/profile/${token}`, GET)
    ↓
[Cloudflare Function] token으로 MySQL 조회 (via Tunnel) → 데이터 반환
    ↓ 응답 수신
[토큰 유효]
    - 이름, 이메일(수정 불가), 스펙 원문, 희망조건 원문을 폼에 pre-fill
[토큰 무효]
    - "유효하지 않은 링크입니다. 등록 이메일을 확인해주세요." 메시지
```

### 폼 필드

| 필드명 | 입력 타입 | 수정 가능 | pre-fill 데이터 |
|--------|-----------|-----------|-----------------|
| 이름 | `<input type="text">` | O | `name` |
| 이메일 | `<input type="email">` | **X** (읽기전용, 회색 배경) | `email` |
| 나의 스펙 | `<textarea>` | O | `raw_spec_text` |
| 희망 조건 | `<textarea>` | O | `raw_pref_text` |

- **이메일이 읽기전용인 이유**: 이메일은 계정 식별 키이므로 변경 불가. 변경이 필요하면 새로 등록해야 한다는 안내 문구 표시.
- **스펙/희망조건**: 처음 입력한 원문(raw text)을 그대로 보여줌. 사용자가 수정 후 재제출하면 Gemini가 다시 파싱.

### 수정 제출 흐름

```
[사용자 수정 완료]
    ↓ [수정하기] 버튼 클릭
[로딩 상태 표시]
    ↓ fetch(`/api/profile/${token}`, POST)
[Cloudflare Function] → Gemini 재파싱 → MySQL UPDATE (via Tunnel)
    ↓
[성공 시]
    - "정보가 수정되었습니다! 다음 공고 분석부터 변경된 정보가 반영됩니다."
[실패 시]
    - 에러 메시지 표시
```

### 추가 기능 (마이페이지 설정)

- **사용자 상태 표시**: 현재 베타 테스터로 전체 기능을 무료로 이용 중임을 표시.
- **알림 수신 설정**: 이메일 알림 수신 여부 선택 (On/Off).
- **탈퇴(수신 중단)**: `[알림 수신 중단 및 데이터 삭제]` 버튼 제공.
- **마지막 수정 일시**: "마지막 정보 수정: 2026-03-10 14:30" 표시

## 4. 구독 결제 페이지 (`/subscribe/:token`)

### 목적

## API 엔드포인트 정리

| 메서드 | 경로 | 역할 | 요청 Body | 응답 |
|--------|------|------|-----------|------|
| POST | `/api/register` | 신규 등록 | `{ name, email, spec_text, pref_text }` | `{ success: true }` 또는 `{ success: false, error: "메시지" }` |
| POST | `/api/send-magic-link` | 간편 로그인 링크 재발송 | `{ email }` | `{ success: true }` (보안을 위해 존재 여부 무관하게 항상 success 반환) |

| POST | `/api/subscribe/confirm` | 결제 확인 (빌링키 발급 + 첫 결제) | `{ customerKey, authKey, plan }` | `{ success: true, subscription }` |

| POST | `/api/webhook/toss` | 토스 웹훅 수신 (결제 성공/실패 알림) | 토스페이먼츠 웹훅 payload | `200 OK` |

## 결제 시스템 설계 (토스페이먼츠)

> Phase 1에서는 구현하지 않음. Phase 5 시점에 구현 예정.
> 그 전까지는 관리자가 수동으로 `is_active`를 관리한다.

### PG사 선택: 토스페이먼츠

| 항목 | 내용 |
|------|------|
| PG사 | 토스페이먼츠 (TossPayments) |
| 결제 방식 | 정기결제 (빌링키 기반 자동결제) |
| 지원 수단 | 신용/체크카드, 토스페이, 네이버페이, 카카오페이 |
| 연동 방식 | 프론트 SDK (결제 위젯) + 백엔드 API (빌링키 발급/결제 실행) |
| 수수료 | 카드 3.3% + VAT (월 1,900원 기준 약 69원) |
| 사업자 요건 | 사업자등록번호 필요 (개인사업자 가능) |

### 선택 이유
- 국내 PG 중 개발자 경험(DX)이 가장 좋음 (문서, SDK, 테스트 환경)
- 사업자등록만으로 가입 가능 (법인 불필요)
- 정기결제 SDK가 잘 되어 있어 직접 카드 정보를 다루지 않아도 됨
- 테스트 모드로 실결제 없이 개발 가능

### 정기결제 흐름 (빌링키 방식)

```
[1. 빌링키 발급 — 최초 1회]
사용자가 결제 수단 등록
    ↓ 토스 SDK: requestBillingAuth()
카드사 본인인증 완료
    ↓ 리다이렉트 + authKey 전달
CF Function → 토스 API: POST /v1/billing/authorizations/issue
    ↓ billingKey 발급
n8n Webhook → MySQL에 billingKey + 결제정보 저장

[2. 자동결제 실행 — 매월 반복]
n8n Schedule (매일 00:00)
    ↓ 결제일 도래한 사용자 조회
토스 API: POST /v1/billing/{billingKey}
    ↓ 결제 성공/실패
MySQL 결제 이력 저장 + is_active 갱신
    ↓ 실패 시
3일간 재시도 (1일 1회) → 3회 실패 시 is_active = 0 + 안내 이메일
```

### DB 추가 테이블 (향후)

```sql
-- 구독 정보
CREATE TABLE subscriptions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    plan_type ENUM('basic', 'premium') NOT NULL DEFAULT 'basic',
    status ENUM('trial', 'active', 'past_due', 'cancelled', 'expired') DEFAULT 'trial',
    billing_key VARCHAR(200),          -- 토스 빌링키 (암호화 저장)
    card_company VARCHAR(50),          -- 카드사명 (표시용)
    card_number_masked VARCHAR(20),    -- 마스킹된 카드번호 ****1234
    current_period_start DATE,
    current_period_end DATE,
    trial_end DATE,                    -- 무료 체험 종료일
    cancelled_at DATETIME,
    created_at DATETIME DEFAULT NOW(),
    updated_at DATETIME DEFAULT NOW() ON UPDATE NOW(),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 결제 이력
CREATE TABLE payments (
    id INT AUTO_INCREMENT PRIMARY KEY,
    subscription_id INT NOT NULL,
    amount INT NOT NULL,               -- 결제 금액 (원)
    status ENUM('success', 'failed', 'refunded') NOT NULL,
    toss_payment_key VARCHAR(200),     -- 토스 결제 키
    failure_reason VARCHAR(500),
    paid_at DATETIME,
    created_at DATETIME DEFAULT NOW(),
    FOREIGN KEY (subscription_id) REFERENCES subscriptions(id)
);
```

### 토스페이먼츠 연동 시 필요한 키

| 키 | 용도 | 저장 위치 |
|----|------|-----------|
| 클라이언트 키 | 프론트 SDK 초기화 | wrangler.toml 환경변수 |
| 시크릿 키 | 백엔드 API 호출 (빌링키 발급, 결제 실행) | wrangler.toml secret + n8n Credential |
| 웹훅 시크릿 | 웹훅 검증용 | CF Function 환경변수 |

### 구독 상태 관리 정책

| 상태 | 조건 | is_active | 알림 발송 |
|------|------|-----------|-----------|
| `trial` | 등록 후 7일 이내, 결제 전 | 1 | O |
| `active` | 결제 성공, 구독 기간 내 | 1 | O |
| `past_due` | 결제 실패, 재시도 중 (3일) | 1 | O (+ 결제 실패 안내) |
| `cancelled` | 사용자 해지, 현재 주기 끝까지 | 1 | O (주기 종료까지) |
| `expired` | 해지 후 주기 만료 / 재시도 3회 실패 | 0 | X |
